`code/model/intergen_housing_fertility/local_panel.py:1038-1076`.

- 4-year persistence: `rho_period = rho_annual**PERIOD_YEARS` (local_panel.py:1045,
  `"rho_period = rho_annual**PERIOD_YEARS"`). With rho_annual=0.9601845894041878 this is
  0.8500000000000001 — exactly the intended 0.85. Checked: rho_annual == 0.85**(1/4) to
  machine zero.
- 4-year innovation: local_panel.py:1049-1051
  `sigma_eps_period = innovation_sd * math.sqrt(sum(rho_annual ** (2 * k) for k in range(int(PERIOD_YEARS))))`
  — the exact time-aggregation `sigma_eps4^2 = sigma_eps1^2 * sum_{k=0..3} rho^{2k}`.
  Value: 0.1216937910487968.
- Unconditional log sd: local_panel.py:1052 `sigma_log = sigma_eps_period / sqrt(1 - rho_period**2)`
  = 0.2310132696147988.
- Recursion (local_panel.py:1053-1063): standard symmetric Rouwenhorst with
  `p = (1+rho_period)/2`, middle rows halved, rows renormalized. Numeric checks: row sums
  exact; stationary eigenvector equals binomial weights `comb(4,k)/16 = (0.0625,0.25,0.375,0.25,0.0625)`
  to 2e-16 (local_panel.py:1064-1065); first-order log autocorrelation = 0.8500000000000002.
- Grid spacing: local_panel.py:1066 `log_grid = linspace(-sigma_log*sqrt(n-1), +sigma_log*sqrt(n-1), n)`
  — standard psi = sigma_log*sqrt(N-1); discrete stationary log variance equals sigma_log^2
  exactly (5.3367e-2 vs theory, diff 2e-17).
- Level normalization: local_panel.py:1067-1068 `z_grid = np.exp(log_grid); z_grid /= float(z_weights @ z_grid)`.
  **What is normalized: the LEVEL grid has stationary mean exactly one under the binomial
  weights** (checked: 0.9999999999999999). The log grid is NOT mean-zero after this shift
  (stationary log mean = -0.02662). Normalization happens once at construction; nothing
  re-normalizes downstream (parameters.py:300-320 only re-normalizes weights/rows, not
  the grid level).
- Resulting grid: z = (0.6135, 0.7729, 0.9737, 1.2268, 1.5456).
- Fingerprint of the constructed objects: `5b6875a240e82073e515a291469e2d259c35bb01ca63cabfb10b639700db5daf`

## 2. What were the "matched" rho/sd matched to? — CONFIRMED reverse-engineering

The matched constants do NOT come from data or from the literature. Numerically exact:

- `MATCHED_ANNUAL_RHO = 0.9601845894041878` == `0.85**0.25` (diff 0.0). 0.85 is
  `DEFAULT_RHO_Z` (local_panel.py:42), the persistence of the OLD ad hoc stay/redraw process.
- The old 5-point grid `[0.60,0.80,1.00,1.20,1.40]` w `[0.1,0.2,0.4,0.2,0.1]`
  (local_panel.py:40-41) has stationary weighted log variance 0.0533671307381197.
  The new process's stationary log variance is 0.05336713073811974 (diff -3.5e-17), and the
  snapshot (`tmp/rouwenhorst_matched_nb120_4h_SOURCE_SNAPSHOT.txt:7-15`) states this was the
  matching target: "matched to the current draft income process's four-year persistence and
  unconditional log variance ... unconditional log variance: 0.0533671307381197".
- So sd_annual = sigma_log_old * sqrt(1-0.85^2) / sqrt(sum rho^{2k}) = 0.06453733259357768.

The paper (`latex/intergenerational_housing_fertility_paper_draft.tex:444-449, 955-956`)
discloses this ("preserving the stationary log-income dispersion of the benchmark process";
"Matched to four-year persistence and stationary log-income dispersion"), so it is not
misrepresented — but the "benchmark process" being matched is the old ad hoc grid, whose
dispersion had no empirical source.

Dispersion vs literature (stationary log variance of the persistent component; literature
values quoted from memory, verify against the papers before citing):

| Process | rho (annual) | innovation sd | stationary log var |

The calibrated persistent-income risk is roughly 4x-12x below standard values (log sd 0.231

## 4. Retirement and pension — VERIFIED

- `J_R = 12` (parameters.py:32): working ages j=0..11 (18-65), retired j=12..16 (66-82);
  first retired age 66.
- Retiree income: `P.income[i,j] = P.pension` for j>=J_R (parameters.py:420); z-independent
  by construction of `income_at_state` (solver.py:129-131: z multiplies income only for
  `j < J_R`). Bellman and KFE both consume `P.income` through `income_at_state`; consistent.
- Pension: balanced stationary PAYG, `tau_pay * avg_worker_income * J_R/(J-J_R)`
  (parameters.py:424-429) = 0.179*1.0*(12/5)*4-year-scale = 1.7184 per period. Uses the
  normalized work-mean profile (=1) and E[z]=1, both exact.
- The z-chain keeps transiting during retirement (Pi_z applied at all ages, solver.py:2096-2099,
  3986), but since retiree income and value no longer depend on z this is inert.
- Old-age 65-75 wealth/income targets: `age_to_index` rounds, so a65s..a75e = j 12,13,14
  (ages 66,70,74) — all retired; the `z=1.0` passed at solver.py:4896
  (`yj = annual_gross_income_at_state(P, i, jj, 1.0)`) is therefore innocuous.

## 6. Leak surfaces for the OLD process — findings

1. `production_profile_overrides()` still ships the OLD grid, weights, stay/redraw Pi_z, and
   persistence 0.85 (production_profile.py:136-141), and `validate_production_profile` does NOT
   validate the income process. The combined spec exists only because the merge order in
   `run_local_panel_case` puts `**income` after `**(extra_overrides)` (local_panel.py:848-853).
   Correct in the overnight (lead reproduced the loss), but fragile: any caller that applies
   profile overrides last, or forgets the income overrides, silently reverts to the old grid
   while still passing profile validation. The recorded `income_process_fingerprint`
   (local_panel.py:1113-1123; in metadata at 607-630) is the only guard.
2. CLI defaults: `cli.py:136-139` — polish defaults `--income-process current`,
   `--income-annual-rho 0.90`, `--income-innovation-sd 0.20`. A CLI reproduction of the
   combined candidate that forgets two flags gets the old grid; one that passes
   `--income-process rouwenhorst` without the rho/sd flags gets the Sommer–Sullivan process
   (4x more dispersed than the matched one). The matched constants are encoded only in
   `code/model/tools/run_intergen_combined_recalibration.py:24-25`,
   `compare_intergen_bequest_normalization.py:39-40`, `build_intergen_mechanics_packet.py:415-420`
   (behind `--combined-corrected-spec`), and the tmp drivers.
3. Checked-in diagnostic/policy tools that hard-default to the OLD process via
   `income_process_overrides(income_states)` with no process argument:
   `tools/run_intergen_policy_poc.py:195`, `tools/audit_intergen_sensitivity_jacobian.py:318`,
   `tools/audit_intergen_solver_accuracy.py:208`, `tools/audit_intergen_mechanism_grid.py:195`,
   `tools/audit_intergen_final_best_pathologies.py:148`, `tools/audit_intergen_parent_credit_margin.py:256`,
   `tools/run_intergen_code_repair_comparison.py:74`,
   `intergen_surrogate_calibration/validate_proposals.py:42`. Running any of these on the
   combined-spec candidate silently evaluates it under the old income process (and without
   q/delta/eta/bequest-normalization fixed spec) — policy numbers would not match the
   estimated model.
4. `income_process_overrides` positional order is (states, process, SD, RHO) — sd before rho
   (local_panel.py:1031-1036). All current call sites pass the arguments in the correct order
   (checked every site listed above), but the (sd, rho) order is a standing footgun.
5. `compare_intergen_income_processes.py` (ANNUAL_RHO=0.90, ANNUAL_INNOVATION_SD=0.20 at
   lines 54-55) documents the Sommer–Sullivan alternative, NOT what was run; anyone reading
   that tool as the provenance of the production process would infer the wrong parameters.

## 7. Legacy denominator quirk (non-markov paths only) — MINOR

`compute_statistics` builds `mean_income` from ENTRY-age income only:
solver.py:4717 `ti = sum(P.income[i, 0] * stats.worker_mass_by_loc[i] for i in range(I))`, and
uses it in `wealth_to_income` / `liquid_wealth_to_income`. With the new profile, entry income is
0.72 of the working mean, so those ratios are overstated ~1.39x in this path. The markov path
overwrites all of them with the correct age-and-z-weighted mean (solver.py:4523-4547), so the
combined-spec run and all its targets are unaffected. Only legacy single-type/permanent-type
diagnostics inherit the distortion.
