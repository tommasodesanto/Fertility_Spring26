Python port of the DT center-periphery lifecycle model

A guided tour of the code in april_26_discrete_time/python_port/dt_cp_model/: what each module does, how the solver flows from a theta vector to moments, and where the speed comes from. This document is a companion to the source — it doesn't duplicate the code, it explains the structure and the non-obvious choices.

What this is

This package is a Python re-implementation of the live MATLAB calibration engine in april_26_discrete_time/run_model_cp_dt.m. The MATLAB code remains the reference; the Python port matches it numerically (max abs moment diff ~0.005 at setup="fast", ~0.003 at setup="benchmark") and runs ~3x faster on the calibration grid because the inner loops are compiled with Numba and partly parallelized.

Conceptually the model is a discrete-time finite-horizon lifecycle with:

  • Two locations (Center vs Periphery; I = 2) with separate prices and rents.
  • Tenure choice across nt = 1 + n_house options: rent at index 0, plus n_house owner-housing rungs of fixed size H_own[k].
  • One-shot completed-fertility over fertile ages: when fertility shocks resolve, all children are conceived together and then mature stochastically.
  • Spatial mobility: a Type-1 EV logit each period over destination locations.
  • Equilibrium: housing markets clear by location through the iterative price + entry-share update.

Data flow

The canonical entry point is

from dt_cp_model import solve_theta
sol, P, p_eq = solve_theta(theta, setup_mode="benchmark")

which threads through this pipeline:

theta (np.array, length 13) │ │ apply_theta(P_base, theta, names) ▼ P (SimpleNamespace of model parameters) │ │ run_model_cp_dt(P) │ │ │ │ loop over GE iterations: │ │ solve_bellman_full / solve_bellman_eval │ │ forward_distribution (fast_stats=True) │ │ update prices p, entry shares e │ │ │ ▼ on accept: │ forward_distribution (fast_stats=False) → full event-study moments ▼ sol (SimpleNamespace) — V, c_pol, hR_pol, bp_pol, tenure_choice, loc_probs, fert_probs, fert_value, g, summary stats P_out, p_eq

The SMM objective at objective.py is a thin wrapper: it calls the same path, then computes a weighted distance between the moments in sol and the live calibration targets.

Model and state space

The household state at age j is a 5-tuple (b, ten, i, n, cs):

dimnamesizemeaning
bliquid wealthNb = 60 (fast) / 80 (benchmark)4-segment grid built in utils.make_grid; spans [b_min, b_max] with denser nodes near zero.
tentenurent = 1 + n_house0 = renter, 1..nt-1 = owner of size H_own[ten-1].
ilocationI = 2Periphery (i=0), Center (i=1).
jageJ = 60Age j+18 in years; retirement at j ≥ J_R.
nparitynpar = 4Number of children ever born (one-shot completed fertility).
cschild stagencs = n_child_stages + 3Encodes how old the cohort of children is. cs = 0 means no children. Higher cs means matured; transitions follow P.Pi_child.

Per-period choices:

  • Savings b' — continuous, solved via 1D golden-section search per cell.
  • Tenure tn ∈ {0, ..., nt-1} — discrete argmax over conditional values.
  • Location i' ∈ {1, ..., I} — Type-1 EV logit with temperature kappa_loc.
  • Fertility at fertile ages j ∈ [A_f_start, A_f_end], parity n' ∈ {0,...,npar-1} — Type-1 EV logit with temperature kappa_fert. One-shot: once you pick n', fertility is over.

Renters consume Cobb-Douglas in (c, h) with weight alpha on consumption; owners consume c and get fixed housing services chi * H_own[ten-1]. Both add a child commitment floor (c̄ + ψ·n, h̄ + h̄_n·n + h̄_jump·𝟙[n≥1]). Period utility is CRRA with curvature sigma.

Bellman recursion

The recursion runs backward in age. At age j the period value is built in four stages, each implemented in solve_bellman_core:

  1. Conditional savings + tenure-conditional value (per i, per ten):
    • Renter (ten=0): pick b' by golden-section to maximize u(c, h) + β · 𝔼V(b', ...), with the renter housing cap h ≤ hR_max binding when desired h would exceed it.
    • Owner (ten≥1): pick b' with the budget R·b + y - oc - c̄ - b' = c, where oc = (δ + τ_H) · p · H_own[ten-1] and housing services h = chi · H_own[ten-1] are deterministic.
    Result: Vd[b, ten, i, n, cs], the value of being in tenure ten at this age before the tenure choice.
  2. Tenure choice: for each origin tenure to, take the discrete max over destination tenures tn. Three economic branches — (i) stay or sell-and-rent (tn = 0), (ii) buy on entry from renting with optional birth grants (to = 0, tn ≥ 1), (iii) sell-and-rebuy (to ≥ 1, tn ≠ to). Infeasibility (below down-payment (1-φ)·hcost or below borrowing limit -φ·hcost) gets -1e10. Result: VH[b, to, id, n, cs], tcj[...].
  3. Location logit: for each origin location io and origin tenure to, compute the value of moving to each destination id. If id ≠ io, we sell and reset to renter at id with sale proceeds heq[io, to] = (1-ψ)·p[io]·H_own[to-1]. Apply logsumexp with temperature kappa_loc and E_loc + move-cost shifters. Result: VI[b, to, io, n, cs], loc_probs[...].
  4. Fertility logit, fertile ages only: at n=0, cs=0, build a value vector across parity choices and logsumexp with temperature kappa_fert. Non-fertile cells: just copy VI into V[..., j, ...].

Tenure choice details

Three move types with non-trivial wealth bookkeeping:

casenet wealth at b'feasibility
to = tn (stay)b directlynone extra
to = 0, tn ≥ 1 (buy)b - hcost (or b - hcost + grant)b ≥ (1-φ)·hcost and b - hcost ≥ -φ·hcost
to ≥ 1, tn = 0 (sell-and-rent)b + heq (clamped to ≥ 0)none extra
to ≥ 1, tn ≠ to, tn ≥ 1 (move-and-rebuy)b + heq - hcostb + heq ≥ (1-φ)·hcost

Two optional birth-related provisions: birth_dp[n, cs, to, tn] (a parity-conditioned down-payment waiver that allows the borrowing constraint to bind instead of the dp threshold) and birth_entry_grant[i, tn, n, cs] (a wealth gift that shifts the savings interpolation point). Both are zero in the live calibration.

Location and fertility logits

Mobility: Type-1 EV with utility V_id + E_loc[id] - move_cost, scaled by 1/kappa_loc, then logsumexp. The closed-form value-of-options is kappa_loc · ls, the conditional probabilities are exp((·) - ls)/Σ(·). Implemented in stable form (subtract the max before exponentiating) in kernels.location_logit_kernel.

Fertility, at fertile ages from a state with n=0, cs=0: a logit over n' ∈ {0, ..., npar-1}, where the value of choosing n' ≥ 1 uses the post-state (n', cs=1). kappa_fert is the temperature.

Bequest and terminal

At j = J - 1 the continuation value is the bequest function

V_bq(b, ten, i, n) = θ_0(n) · ((b + p[i]·H_own[ten-1] + θ_1) ^ (1-σ) - 1) / (1-σ)

with θ_0(n) = 1 + θ_n · n. Lifetime housing is liquidated at the local price.

Backward induction (the j loop)

Implemented in solver.solve_bellman_core. For each age j from J-1 down to 0:

for j in range(J - 1, -1, -1):
    Vc = apply_child_aging(V[..., j+1, ...] or V_bq)   # advance child-state
    for i in range(I):
        if eval_mode:
            renter block (kernel)    # plug in stored bp, no optimization
            owner blocks (kernel) for ten in 1..nt-1
        else:
            renter block (kernel)    # full Bellman: golden-section
            owner blocks (kernel) for ten in 1..nt-1

    VH, tcj = tenure_choice_kernel(Vd, ...)
    VI, lpj = location_logit_kernel(VH, ...)
    if in_fert: fertility logit
    V[..., j, ...] = ...   # write back

The four kernels here are each a Numba @njit(parallel=True) function with prange(nc) over the flattened (n, cs) column index. Each c writes to a disjoint output cell, so no synchronization is needed.

F-order convention. The (n, cs) pair is flattened to a 1-D column index c = n + npar·cs (parity varies fastest) before entering kernels, to match the MATLAB column-major reference. utils.flat_nc and utils.unflat_nc do the reshape.

Howard policy iteration

Solving the Bellman from scratch every GE iteration is wasteful — once we have a sensible savings policy bp_pol, the next GE step's Bellman is well-approximated by re-evaluating utility at bp_pol instead of re-running golden-section. run_model_cp_dt implements this:

do_full = force_full or it ≤ 3 or stored_bp is None
       or (it % howard_freq == 0)
       or stall_count ≥ 2

if do_full:
    full Bellman, store bp
else:
    eval Bellman, plug in stored bp

The eval branch is much cheaper: each (b, c) cell does one continuation-value interpolation and one closed-form CD utility evaluation, instead of golden-section. The two paths share the same post-Bellman pipeline (tenure, location, fertility), so savings are concentrated in the inner loops, which is exactly what the eval_*_block_kernel kernels target.

Forward distribution

Two paths, in solver.forward_distribution:

  • Fast-stat (compiled, kernels.forward_distribution_fast_kernel) — propagates only what the GE loop needs (population shares, housing demand, total births, mature entrants). Used inside every equilibrium iteration.
  • Full-stat (Python) — recomputes event-study housing moments (housing_increment_0to1_eventstudy_t3, 1to2_proxy_t3) at the accepted equilibrium only. These need extra mass tracking by birth-event time and aren't worth doing inside the GE iteration.

Each step of the forward pass advances one age:

  1. Fertility: at fertile ages, multiply mass at (n=0, cs=0) by fert_probs and split into (n', cs=1) bins.
  2. Location: each cell splits across id ∈ {1..I} by loc_probs. Movers inherit the renter state and have wealth b' = clip(b + heq[io, to], b_grid).
  3. Tenure: tn = tcj[b, to, id, n, cs] deterministically (already an argmax).
  4. Savings: deposit mass on the two grid-neighbors of bp_pol with linear weights.
  5. Child aging: matrix-multiply by P.Pi_child[:, :, n].

GE loop

Implemented in solver.run_model_cp_dt. State variables: housing prices p ∈ ℝ^I, entry shares e ∈ ℝ^I (sum to 1). Each iteration:

  1. Compute r = user_cost_rate · p.
  2. Solve Bellman (full or eval, see Howard).
  3. Forward distribution (fast-stat).
  4. Update p from housing-market clearing: excess demand drives prices up. Clamp updates to [lambda_price_min, lambda_price_max].
  5. Update entry shares from observed mature entrants by location, with floor and damping.
  6. Compute equilibrium error (price + entry-share residual). If small enough, accept.

The exit ladder is tiered:

  • Hard convergence: err < tol_eq.
  • Near-converged: it > 10 and best_err ≤ tol_eq (best across iterations, not necessarily current).
  • Soft-accept: it > 25 and best_err < 10·tol_eq.
  • Cap: it > 50 and best_err < 0.02.

The reason for the soft tiers: small-grid models with discrete tenure choice can oscillate around tenure thresholds rather than converge, so a strict err < tol_eq exit isn't always achievable. This matches the live MATLAB solver.

Numba kernels

kernels.py holds the compiled inner loops. The kernels split into three groups:

Bellman block kernels

kernelused byshape of work
full_renter_block_kernelfull Bellman, renterper (i, j), loops (c, b) with golden-section + CD post-search
full_owner_block_kernelfull Bellman, ownerper (i, j, ten), loops (c, b) with golden-section
eval_renter_block_kernelHoward eval, renterper (i, j), loops (c, b) with closed-form CD evaluation
eval_owner_block_kernelHoward eval, ownerper (i, j, ten), loops (c, b) with closed-form owner evaluation

All four are @njit(cache=True, parallel=True) with for c in prange(nc). Race-free because each iteration writes (*, c) only.

Post-Bellman per-j kernels

kernelrole
tenure_choice_kernelDiscrete max over tn with feasibility masking; produces VH and tcj.
location_logit_kernelStable logsumexp across destination locations; produces VI and lpj.

These are @njit(cache=True) (not parallel — outer loops are tiny, I·nt = 8-10).

Forward-distribution kernel

forward_distribution_fast_kernel drives one full forward pass. It dispatches into _advance_mass_component for the per-cell tenure→savings→aging step. Single-threaded because the mass updates are accumulating writes (g[...] += ...) which would race; we deliberately keep a sequential scatter pattern.

Tiny utility kernels

  • interp_scalar, _interp_with_clip — 1D linear interpolation with bounds clipping (binary search).
  • interp_cols_kernel, interp_cols_preidx_kernel — column-wise interpolation; the preidx variant uses cached (idx, wt).
  • scatter_vec_kernel, scatter_cols_kernel, scatter_cols_sameidx_kernel — mass-preserving scatter-add for the (less hot) full-stat distribution path.
  • eval_renter_scalar, eval_owner_scalar — per-b' period-utility-plus-continuation evaluators called inside the golden-section loops.

Parallelism

The four block kernels parallelize over the flat column index c, which has nc = npar·ncs = 24-28 elements. Each iteration of the c loop touches its own slice of the output arrays (Vo[:, c], bp_out[:, c], etc.), so adding parallel=True + prange is safe without atomics or barriers.

Why we don't get ideal scaling on an 8-core machine:

  • prange(24) divides poorly across 8 threads (3 columns each, then thread-spawn / join overhead).
  • The eval-mode kernels do little compute per (b, c) cell — the closed-form CD evaluation is ~10 FLOPs. Memory bandwidth, not compute, is the ceiling.
  • The tenure-choice and location-logit kernels are still single-threaded.
  • The forward_distribution path is single-threaded.
  • The GE loop itself is sequential.

Realised gain on this 8-core laptop: about 1.35x on top of the serial port, measured at setup="benchmark" direct-theta. CPU usage averages 165–170% of wall during a run.

SMM objective

objective.smm_objective_dt wraps the model path:

theta → apply_theta(P_base) → run_model_cp_dt → moments → weighted SSE

Targets and weights live in parameters.build_calibration_setup. The objective handles three concerns:

  1. Bounds checking on theta. Out-of-bounds returns 1e6.
  2. Optional geography inversion (see below).
  3. Aggregation of model-vs-target deviations: loss = Σ_t weights[t] · ((model[t] - target[t]) / |target[t]|)², plus an optional inversion penalty.

Geography inversion

The two location-supply wedges E_loc[Center] and r_bar[Center] are not directly identified by individual moments — they map onto observed center population share and rent ratio. The inversion loop, when enabled, adjusts these two scalars to hit those two aggregate targets:

for inv_iter in 1..max_iter:
    solve model at current (E_C, r_bar_C)
    if |pop_share_C - target| and |rent_ratio - target| are small enough: break
    E_C       += kappa_loc · speed · log(target_odds / model_odds)
    r_bar_C   *= (rent_ratio_target / model_rent_ratio) ^ speed

This is an "outer loop around the GE loop". For pure model exploration (e.g. the gradient-descent script in tools/gradient_descent.py) the inversion is skipped and (E_C, r_bar_C) are taken as exogenous, which gets you a single-pass evaluation and a much smoother objective.

File-by-file reference

evaluate.py

Holds the canonical entry point.

def solve_theta(theta, setup_mode="fast", max_iter_eq=None, verbose=False)

Builds P_base via build_calibration_setup(setup_mode), applies theta with apply_theta, optionally overrides max_iter_eq, and returns run_model_cp_dt(P). Nothing else here — the SMM path uses the same chain.

theta.py

Defines the parameter vector convention. PARAM_NAMES lists 13 names in order; theta_bounds(theta) returns box bounds; apply_theta(P, theta, names) writes the values into P and recomputes derived quantities (e.g. rho). The SMM and direct-theta paths both go through this function.

parameters.py

Three responsibilities:

  • setup_parameters() — defaults for one fixed setup.
  • build_calibration_setup(setup_mode) — returns a struct with P_base, names, x0, targets, weights, inversion_targets. Two modes: "fast" (Nb=60, lower tolerance) and "benchmark" (Nb=80, the calibration grid).
  • Helpers for child-state transitions (make_child_transition_matrix_with_matured) and theta application internals.

solver.py

The big file (~1800 lines). Public functions:

  • run_model_cp_dt(P, verbose) — GE loop, returns (sol, P_out, p_eq).
  • solve_bellman_full(r, p, P, b_grid, SD) — one full backward induction.
  • solve_bellman_eval(stored_bp, r, p, P, b_grid, SD) — Howard evaluation.
  • forward_distribution(bp, hR, tc, lp, fp, r, p, P, b_grid, SD, fast_stats) — one forward pass.
  • extract_moments_from_dist(g, P, b_grid, ...) — population, ownership, fertility, housing summaries.

Internal: solve_bellman_core is the shared backward induction with a flag for full vs eval mode. apply_child_aging is one matmul per parity (BLAS, fast).

kernels.py

See the Numba kernels section.

objective.py

SMM wrapper. Used by cli.py's objective command and by the gradient-descent driver.

utils.py

Numerical primitives: make_grid, interp_indices, interp_vector, interp_on_grid, interp_cols, scatter_redistribute, logsumexp, flat_nc / unflat_nc, weighted_quantile.

cli.py

Three subcommands:

  • solve-theta --setup <mode> --theta <x0|csv> — direct-theta solve, writes JSON.
  • benchmark --setup <mode> --max-iter-eq <N> [--force-full] — timing benchmark.
  • objective — SMM loss at x0.

Gotchas

  • phi is the financed share, not the down payment. So the down-payment threshold is (1 - phi) · hcost and the borrowing limit is -phi · hcost. Easy to flip.
  • F-order flattening. The flat column index c obeys c = n + npar · cs (parity varies fastest). All flat_nc / unflat_nc reshapes use order="F".
  • Soft equilibrium accept is intentional. err < tol_eq isn't always reachable on small grids with discrete tenure choice; the tiered exit ladder mirrors the MATLAB reference and prevents wasted iterations.
  • Howard eval reuses bp_pol from the most recent full Bellman. If you change anything that invalidates that policy (e.g. different price p in the same call), the eval pass will give wrong answers. run_model_cp_dt guards against this by always doing the first 3 GE iterations as full Bellman.
  • setup="fast" is for parity tests, not calibration. All calibration speed numbers should be at setup="benchmark". The ~10x-vs-MATLAB speedup that earlier benchmark JSONs report was at the small grid; the real-grid number is ~3x.
  • Numba caches. Each kernel signature compiles once and stores the compiled object under dt_cp_model/__pycache__. Touching the kernel source invalidates the cache and the next call recompiles (~5–10 seconds per kernel). The parallel=True form has its own cached binary separate from the serial form.
  • Bridge benchmark theta lives in tools/check_bridge_fit.py and tools/gradient_descent.py. If the live calibration anchor changes, those constants need to be re-extracted from the saved .mat file; there's no automatic link.
